# /money-plan — 行动方案（小成本先验证）

把一个机会变成"今天就能动手 + 能低成本验证真能赚到钱"的计划。

## 跨平台说明（一次性，下文不再重复）

下文"网页检索/打开网页"：Claude Code 用 WebSearch / WebFetch，Codex 用其内置等价能力。`allowed-tools` 是 Claude Code 权限声明，Codex 忽略。

## 前置检查
- 读 `.money-state.json` 找该机会。若未验证过或判定为高危 ❌，**先劝用户走 `money-verify`**，不直接做计划。
- 存疑 ⚠️ 的可以做计划，但计划必须以"验证能不能收到第一笔钱"为第一目标。

## 方案必含 5 块

1. **验证第一步（最重要）**：一个最小动作，目标是用尽量少的时间/钱确认"真的有人会为此付钱"。
   例：先在闲鱼挂一个最小服务、先发 3 条内容看数据、先接 1 单最小的活。**未验证前不投入大量时间。**
2. **前两周动作清单**：具体到"每天/每周做什么"，可量化（发几条、投几个、接几单）。
3. **诚实收入预期**：前期大概多久见第一笔钱、金额区间。明说这是"时间换小钱+攒作品"还是有快钱。不画饼。
4. **止损线**：满足什么条件就该停（例：投入 X 小时后零正反馈 / 出现任何反诈红线 / 要求付费）。
5. **复用 AI 提效点**：这个机会里 AI 具体帮你提效在哪一步（别让"用 AI"停在口号）。

> 需要时用网页检索查该机会当下的具体操作路径（平台怎么开通、怎么接单、怎么提现）。

## 落盘（含「预期」——这是校准环的起点）
把方案写入 `.money-state.json` 的 `active`（chosen_id / started_at=今天 / plan），机会 status 改为 `chosen`。
**同时写下 `active.prediction`**：预计投入多少小时、多久见第一笔钱、第一笔收入区间（recorded_at=今天）。
这份预期写完就别改——日后 money-retro 拿"实际"和它对账，才能校准这套方法准不准。

## 收尾
提示：开始执行后用 `money-status` 看进度；到了验证节点或 T+N 天用 `money-retro` 复盘实际投入与收入。
再次强调止损线和反诈红线——一旦对方开始要钱，立刻停。
